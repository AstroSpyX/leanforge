2026-05-24 — v1.1.0 — tool-use + Google Gemini provider; llm/ reorganized
-------

  Scope: bundled architecture change on branch `tooluse-gemini`.
  Hard cutover from prose-extraction to native tool-use, with a
  second provider (Google Gemini via `google-genai`) added at the
  same time so the abstraction is exercised by two implementations
  from day one.

  Why now:
    Bugs 11, 12, 13 (truncation, remaining_blockers int-vs-string,
    Range shape malformation) all traced to the same root cause —
    parsing structured intent from a free-text response. Tool use
    with `strict: true` (Anthropic) and Gemini's schema-driven
    function calling validate the model's output server-side before
    we ever see it, so that whole class of failures becomes
    impossible by construction.

  Architecture reorg (llm/):
    llm/
    ├── __init__.py            — re-exports public API + .env load
    ├── ask.py                 — provider-agnostic dispatcher
    ├── errors.py              — typed AskLLMError hierarchy
    ├── response.py            — Response + ToolCall dataclasses
    ├── models.py              — registry, now with 6 models
    ├── cache.py               — disk cache (unchanged)
    ├── tools/
    │   ├── __init__.py        — ToolSpec dataclass
    │   └── submit_fix.py      — Pydantic-derived submit_fix tool
    ├── providers/
    │   ├── anthropic.py       — Anthropic adapter (strict=true)
    │   └── google.py          — Gemini adapter (google-genai SDK)
    └── tests/                  — full coverage of every module

  Tool-use cutover:
    - `llm.ask(tools=[...], tool_choice="submit_fix", ...)` returns
      a Response with `tool_calls: list[ToolCall]`. Strict mode
      guarantees `tool_calls[0].input` matches the schema.
    - submit_fix tool is derived from RefineResponse.model_json_
      schema(), then post-processed: inline $refs (Anthropic strict
      forbids them), set additionalProperties:false on every object,
      strip $schema.
    - Controller now calls ask() + reads response.tool_calls[0].
      _extract_json_object and _parse_response_or_fail deleted.
    - prefill parameter removed everywhere (was always None per
      v1.0.1; tool use renders it irrelevant).
    - Bug-12 field_validator on RefineResponse.remaining_blockers
      deleted as dead code (strict mode makes coercion unreachable).

  Gemini provider:
    - Uses official `google-genai` SDK (NOT the deprecated
      `google-generativeai`).
    - Three models registered: gemini-flash-lite, gemini-flash,
      gemini-pro (stable internal keys; provider_model_id can bump
      independently as Google releases new versions).
    - GOOGLE_API_KEY added to .env.example.
    - Tool/tool_choice translation: ToolSpec → types.Tool with
      function_declarations; tool_choice → types.ToolConfig with
      AUTO/ANY/allowed_function_names.

  Deleted code:
    - llm/ask_llm.py (replaced by llm/ask.py + providers/*)
    - refine/controller._extract_json_object
    - refine/controller._parse_response_or_fail
    - refine/prompt_builder.ASSISTANT_PREFILL
    - prefill from build_repair_messages / build_generation_messages
      return tuples (now 3-tuples instead of 4-tuples)
    - RefineResponse._coerce_blockers_to_str validator
    - SYSTEM_PROMPTS dict, `mode` parameter, ask_llm_text /
      ask_llm_fix helpers (all unused outside their definitions)

  System-prompt cleanup:
    Both repair and generation prompts shrank by ~40% — the schema
    section in prose is gone (schema is enforced by tool spec).
    What remains: edit rules, scope contract, what's allowed,
    repair-strategy enum values. Bumped PROMPT_VERSION to repair_v2
    / generate_v2.

  QA: 367 tests pass, 1 skipped (live-API smoke). ruff + mypy strict
  clean. Mypy now runs on full llm/ + refine/ source (was previously
  invoked file-by-file). New per-file ignore for leanforge.py and
  smoke_test.py (long doctring example lines, unrelated to refactor).

  Files added:
    llm/ask.py, llm/errors.py, llm/response.py, llm/tools/__init__.py,
    llm/tools/submit_fix.py, llm/providers/__init__.py,
    llm/providers/anthropic.py, llm/providers/google.py,
    llm/tests/{test_ask, test_response, test_tools, test_submit_fix}.py,
    llm/tests/providers/{test_anthropic, test_google}.py.

  Files modified:
    llm/__init__.py, llm/models.py, llm/smoke_anthropic.py,
    refine/controller.py, refine/prompt_builder.py, refine/schema.py,
    refine/tests/{test_controller, test_prompt_builder, test_schema}.py,
    pyproject.toml, .env.example, CLAUDE.md, RUN_COMMANDS.txt,
    IMPROVEMENTS.txt.

  Files deleted:
    llm/ask_llm.py.

  Next: OpenAI provider is now a ~150-LOC drop-in (

2026-05-24 — milestone — Group.lean Stage 7 (quotient group) proven by Haiku in 1 iter for $0.022
-------

  Test: 4 sorries in Stage 7 — quotient_mul (nested Quot.lift),
  quotient_mul_well_defined (uses normality to handle conjugation),
  quotient_inverse (Quot.lift over inv), quotient_is_group (assembles
  the MyGroup record on MyQuotient). Stages 8-9 in block-comment guard.
  Model switched to Haiku 4.5 (--model haiku) after Anthropic funds
  ran low mid-session.

  Result: SUCCESS in 1 iter, $0.022. 50-100x cheaper than the
  Sonnet equivalent would have been.

  Proof highlights:
    - quotient_mul: nested Quot.lift (outer over a, inner over b),
      with well-definedness witnesses via Quot.sound +
      quotient_mul_well_defined. Outer well-definedness uses
      Quot.inductionOn over the second argument.
    - quotient_mul_well_defined: 7-line calc-style rewrite chain
      manipulating Γ.op (Γ.op (Γ.inv b) (Γ.op (Γ.inv a) a')) b,
      using the conjugation form of normality. Non-trivial algebra.
    - quotient_is_group: full MyGroup record with all 7 fields.
      Each axiom proof uses Quot.inductionOn to reduce to the
      underlying group's axiom + a quotient_rel reflexivity step.

  Notable agent decision, unprompted:
    Haiku REORDERED the declarations. Stubs were placed as
    quotient_mul then quotient_mul_well_defined, but the former
    references the latter. Haiku silently swapped them in the
    output so the dependency order is correct.

  Cost calibration update:
    Stage 1     ~$0.008
    Stage 1+2   ~$0.10
    Stage 3+4   $0.46 (Sonnet, 9 iters, 10 sorries)
    Stage 5+6   $2.04 (Sonnet, 3 attempts, 11 sorries)
    Stage 7     $0.022 (Haiku, 1 iter, 4 sorries) ← new floor
    Per-sorry:  $0.04 Sonnet (Stage 3+4) → $0.0055 Haiku (Stage 7)

  Caveat: Haiku worked because the prompt was very detailed —
  named Quot primitives explicitly (Quot.mk, Quot.lift, Quot.sound,
  Quot.ind), called out which prior decls to lean on. Haiku is more
  prompt-quality-dependent than Sonnet; this run benefited from
  iterating on the goal text in previous stages.

  Files: examples/scratch/Scratch/Group.lean (Stage 7 proven;
         Stages 8-9 still in block-comment guard).

2026-05-24 — v1.0.8 — Bug 12 fix: remaining_blockers schema coercion + prompt clarification
-------

  Test: phased run on Stage 5+6 (11 sorries). Iter 4 died with:
  "9 validation errors for RefineResponse / remaining_blockers.0
   Input should be a valid string [type=string_type, input_value=2,
   input_type=int]" — Sonnet emitted [2, 4, 5, ...] for
  remaining_blockers.

  Bug 12 — schema/prompt mismatch on remaining_blockers.
    Prompt says "list its `id` in remaining_blockers" while id is
    explicitly an int everywhere else (Fix.diagnostic_ids: list[int]).
    Pydantic schema requires list[str]. The model followed the natural
    reading and used the int form, failing validation.
    Fix: @field_validator('remaining_blockers', mode='before') on
    RefineResponse coerces int → str list-wise. Prompt updated in two
    places to say "list its id as STRING (e.g. \"2\")". Belt-and-
    suspenders: prompt nudges correct behavior; schema is tolerant.

  Files: refine/schema.py (validator), refine/prompt_builder.py
         (two prompt sites), refine/tests/test_schema.py (two new tests
         covering pure-int and mixed int/str coercion).

  QA: 342 tests, ruff + mypy --strict clean.

  Related findings (not fixed yet — see IMPROVEMENTS.txt):
    - LLM_ERROR is fatal across entire run. A single malformed
      response kills a loop that was making forward progress
      (iter 3 had gone 22 → 12 errors). Should at minimum offer
      one corrective-retry attempt.
    - Cost tracking drops on failed iters. The user-facing
      "cost=$X" line at end of run only sums iter cumulative_cost,
      but failed iters never write IterationState, so the API
      call's tokens are invisible.

2026-05-24 — discovery — strategy ceiling at ~10 sorries; Anthropic timeout cascade
-------

  Test: Stages 5-9 dumped into one run (21 sorries on a ~600-line
  file). max_tokens=32768, max_iters=50, max_cost=$5.00.

  What happened:
    iter 1: regression. Sonnet's "rewrite everything in one shot"
            strategy that worked at 7-10 sorries did NOT scale to 21.
            Resolved 20 sorries but introduced 37 new errors; net
            regression. Cost $0.18.
    iter 2: LLM_ERROR. The cleanup prompt was now enormous (broken
            874-line file + 36 diagnostics + recent-failures memo).
            Sonnet tried another big generation. Anthropic's server-
            side timeout for very-long-generation requests fired.
            SDK auto-retried 5 times; each hit the same wall.
            Final: "Request timed out or interrupted."

  Bug C (revisited) — model-strategy ceiling.
    The agent's "fix it all in one response" approach has a
    measurable upper bound around 10-11 sorries on Sonnet 4.6.
    Below that, single-shot is the cheapest path. Above that, the
    accumulated errors in the rewrite exceed what the cleanup loop
    can recover from before the next iter's prompt becomes too big
    to process. Not a code bug — a strategy bug surfaced by scale.

  Bug D (new) — Anthropic server-side request timeout cascade.
    When max_tokens is high (32K) and the prompt is also large
    (broken file + many diagnostics + memo), Anthropic's API has
    an upper bound on the time it'll spend generating before
    returning an error. We get "Request timed out or interrupted."
    The SDK auto-retries 5 times by default but each retry hits
    the same wall. We have no graceful-degradation path: we can't
    say "this time generate less" mid-retry.

  Recovery: reverted to Stage 5+6 only (11 sorries) phased run.
  That run uncovered Bug 12 (above) and validated the strategy:
  smaller-scope runs land cleanly; bigger ones cascade.

  Forward decision: pause on big-scope runs until either
    (a) --sorries-per-iter N CLI flag lands (forces small per-iter
        response shape), OR
    (b) tool-use migration (v1.1.0) lands (Bug 12 disappears
        structurally; opens door to per-tool-call scoping).

  Files: no code change in this entry — discovery + design
         direction. The Stage 5+6 phased run trace lives at
         examples/scratch/.refine/Scratch__Group.lean.archive-*.

2026-05-23 — v1.0.5 — archive prior-run state on each new run
-------

  Test: ran refine on Group.lean three times in sequence. Symptoms
  surfaced during multi-run use:
    - history.jsonl interleaved iterations from all three runs in
      one file (iter 0..4 from run 1, iter 0..1 from run 2, iter 0
      from run 3)
    - iter_NNNN.lean snapshots from prior runs were silently
      overwritten when a later run reached that iteration number
    - orphan snapshots from longer prior runs stuck around with
      no context

  Bug 9 — refine state was global per file, not per run.
    Multi-run produced comingled history and silent overwrite of
    prior snapshots. Made history.jsonl unreadable as a per-run

    Past runs preserved intact as sibling directories; current run
    starts clean. Implemented as new ArtifactStore.start_new_run()
    method replacing the bare initialize() call at refine() start.

  Files: refine/artifact_store.py (new start_new_run +
         _allocate_archive_path), refine/controller.py (call
         start_new_run, log archive path), refine/tests/test_artifact_store.py.

    to the LLM. Recent-failures memo carried nothing useful.
    Fix: capture the exception message into the iteration's
    soft_warnings; _describe_rejection reads soft_warnings as a
    fallback so the memo actually tells the LLM what went wrong.

  Bug 8 — redundant "subtract 1" instruction in system prompt.
    Removed now that display matches JSON convention. Also reworded
    EDIT RULES for clarity.

-------
