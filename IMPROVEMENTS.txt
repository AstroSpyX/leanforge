--------------------------------------------------------------------------------
CORRECTNESS / ROBUSTNESS
--------------------------------------------------------------------------------

  * Stuck-loop defenses (deterministic-temp + cache + identical
    prompt = unbreakable; surfaced by Phase 3 run on 2026-05-24).
    The framework currently lets the loop spin indefinitely (up to
    max-iters) when temp=0 + identical prompts produce identical
    broken responses. Layered defenses, in priority order:
      1. Don't cache responses that produced Lean errors.
         Infrastructure for the other defenses; alone does NOT help
         since temp=0 makes fresh calls equally broken. (~30 min)
      2. Verify + fix loop detection. Controller has
         LOOP_DETECTION_WINDOW=5 and computes state_hash but
         apparently didn't fire on the 5-iter repeat. Audit. (~1 hr)
      3. Stronger recent-failures memo with explicit "you've tried
         X N times — STOP" framing. Changes prompt → changes
         response. (~2 hrs)
      4. Temperature bump on detected repeat (0 → 0.5 on 2+
         consecutive identical responses). Requires #1. (~1 hr)
      5. Auto-escalate to a stronger model on detected repeat for
         a "rescue iter" or two. (~3 hrs)
      6. Pause-fallback in auto mode — pause.py infrastructure
         exists but only wired into mode=step. (~half day)
    None implemented yet; Phase 3 is the first empirical evidence
    the failure mode is real and matters.

  * Don't make LLM_ERROR fatal — offer at least one corrective retry.
      Observed: Stage 5+6 phased run (2026-05-24). Iter 3 was making
      forward progress (errors 22 → 12), iter 4's LLM response failed
      Pydantic validation (Bug 12), controller threw AskLLMError and
      the loop quit immediately. No retry. Suggested fix: on
      AskLLMError that's a schema-validation failure, retry once with
      a corrective system-prompt addendum ("your previous response
      had X type error; emit Y instead"). Only escalate to fatal on
      the second failure. Transport-layer errors (timeout, rate
      limit) the SDK already retries; this would cover the
      MODEL-OUTPUT-MALFORMED class specifically.

  * Cost tracking on failed iters.
      Observed: same run. iter 4 burned ~$0.10 of API budget on a
      malformed response, but cost=$X at end-of-run summary only sums
      IterationState rows — which failed iters never produce. The
      cumulative cost line under-reports actual spend.
      Suggested fix: ask_llm tracks tokens-charged for ALL calls, not
      just successful ones; controller writes a sidecar
      `failed_calls.jsonl` capturing tokens + cost for any non-200
      that still consumed quota.

  * Model-strategy ceiling at ~10 sorries (single-shot rewrite).
      Observed: Stage 5-9 dump on 2026-05-24 (21 sorries).
      Sonnet 4.6 hits a measurable ceiling around 10-11 sorries
      per single response. Above that, accumulated mistakes in the
      rewrite exceed what the cleanup loop can recover before the
      next prompt becomes too large for Anthropic's
      generation-time budget. Two mitigations: (a) `--sorries-per-
      iter N` CLI flag (see UX section); (b) tool-use migration
      (already DECIDED for v1.1.0) lets us split submit_fix into
      per-fix calls. Until one lands, runs >10 sorries are
      fragile.

  * Anthropic generation-time-budget timeout cascade.
      Observed: same Stage 5-9 dump. When max_tokens is high (32K)
      and the prompt is large (broken file + many diagnostics +
      memo), Anthropic returns "Request timed out or interrupted"
      and the SDK auto-retries — same input, same wall, 5 retries
      all fail. We have no path to say "retry with smaller
      max_tokens / fewer diagnostics shown / abbreviated memo."
      Suggested fix: on the SDK's timeout-with-retries-exhausted,
      controller drops max_tokens by half and retries the same
      iter once before exiting with LLM_ERROR.

  * Prune unused `have` statements in final proofs.
      Observed: AmGm.lean v1.0.4 run left `have hxy : 0 < x * y := mul_pos
      hx hy` unused — Claude added it on a failed attempt and the
      survivor proof didn't consume it. Add a final cleanup pass that
      asks Claude to tighten the proof once SUCCESS is reached, or
      detect unused hypotheses via Lean's unused-variable linter.

  * Mathlib lemma name drift detection.
      Observed: Claude tried `le_div_iff` (the pre-Mathlib-v4 name),
      hit ELAB_ERROR, recovered to `le_div_iff₀` on retry. Cost: 1
      wasted iteration. Could pre-supply a curated alias map in the
      system prompt, OR run `Lean Search` / `loogle` on hypothesized
      lemma names before sending them to Claude.

  * Theorem-signature preservation: stronger semantic equivalence check.
      Currently text-level (sig_hash). A determined LLM can weaken a
      theorem by introducing a redefinition (`def BadTrue := True;
      theorem foo : BadTrue := trivial`). v2 work per spec.

  * AST-aware pre-validation.
      Run a syntax-only parse on the modified file BEFORE invoking full
      elaboration. Catches obvious bracket/paren mistakes in <100 ms
      vs the 10-30 s of a full Mathlib elaboration. Listed as v1.5 in
      spec.


--------------------------------------------------------------------------------
UX / DX
--------------------------------------------------------------------------------

  * `--sorries-per-iter N` CLI flag.
      Forces the LLM to a bounded scope per response, preventing the
      "rewrite everything in one shot" strategy that hits the 10-
      sorry ceiling. When set, prompt is modified to add: "Choose at
      most N sorries to fix this iteration; pick those whose proofs
      can be given in under 10 lines; leave others as sorry." The
      controller validates the response respects the cap. Tradeoff:
      total cost is higher (N+ iterations vs 1-3) but failure
      modes are bounded — no more catastrophic regressions or
      timeout cascades. Default unbounded (current behavior). Worth
      pairing with selection-guidance: "prefer sorries whose
      dependencies are already proven."

  * Pause / step mode wiring.
      pause.py is fully implemented (c/r/n/a/q/Q/s/d/h/g/b menu).
      controller.py only dispatches `auto` mode. Adding policy + step
      is a small switch in the loop body — call prompt_user() when the
      controller decides to pause and act on the PauseChoice.

  * `--no-cache` and `--clear-cache` CLI flags.
      Currently: `rm -rf llm/.cache` between runs. Inconvenient and
      easy to forget. --no-cache bypasses the cache for one run;
      --clear-cache wipes it before starting.

  * `--purge-archives [DURATION]` CLI flag.
      v1.0.5 added per-run archive directories
      (.refine/<slug>.archive-<timestamp>/). Over many runs these
      accumulate. A `--purge-archives 7d` flag would delete archives
      older than the given duration; no arg = nuke all archives.
      Currently the manual workaround is `rm -rf .refine/*.archive-*`.

  * `--show-diff` flag.
      At end of run, print `diff iter_0000.lean <final>` so the user
      sees what changed without a separate cat/diff invocation.

  * Open `.refine/` folder automatically after run.
      Optional --open flag, macOS-only. Useful when running interactively.

  * Better terminal output formatting.
      The current stderr stream is a flat plain-text log. Colors for
      status (green=success, yellow=progress, red=regression), boxed
      status block at the end. ANSI escape codes; gated on isatty().

  * Cost preview before each call.
      Print estimated tokens + cost before sending. Helps catch
      runaway-prompt scenarios early. Already have the cost table.

  * Strategy-nudge auto-suggest at pause.
      When pausing on CHURN or NO_CHANGE, suggest a strategy that
      hasn't been tried yet ("tried tactic_rewrite × 2; consider
      type_annotation_fix?"). Cheap heuristic.


--------------------------------------------------------------------------------
PERFORMANCE / COST
--------------------------------------------------------------------------------

  * Declaration slicing — promote from v1.5 to v1.1.
      Spec rev 4.1 cost note: ">$0.05/iter on files >500 lines" is the
      trigger to promote. AmGm.lean was 4 lines so it didn't trigger;
      next medium-file run probably will. Send focused decl + immediate
      neighbors instead of full file. Likely 5-10x cost reduction on
      large files.

  * Anthropic prompt-cache activation.
      Currently `cache_prompt=True` but the system prompt for refine
      is below the 1024-token minimum block size, so caching never
      activates. Either pad the system prompt to >1024 tokens (with
      useful content, not filler), or include the file content in the
      cached block (more complex but higher hit rate).

  * Parallel candidate generation (v2).
      Ask N candidates per iteration at temp > 0, pick best by error
      count + fingerprint diversity. Immutable snapshots already support
      this. Architecture-ready; needs the orchestration.

  * Beam search across iterations (v2).
      Maintain top-K candidate branches. Combine with parallel candidates.


--------------------------------------------------------------------------------
DIAGNOSTIC / OBSERVABILITY
--------------------------------------------------------------------------------

  * Live token usage display.
      Stream the running token count to stderr during long generations.
      Anthropic SDK has streaming tokens; we currently use stream=false.

  * Goal-state evolution visualization.
      Render the goal text from each iteration side by side, highlight
      diffs. Useful for "did my proof actually approach the goal" review.

  * History.jsonl human viewer.
      A small CLI like `refine inspect <project> <file>` that formats
      history.jsonl as a readable timeline. The Python one-liners in
      RUN_COMMANDS.txt are functional but not pretty.

  * Optional events.jsonl population.
      events.jsonl is committed-to in the spec (controller event log)
      but the controller doesn't actually write to it yet. Wire up
      pause / user_choice / budget_raised / loop_end events.


--------------------------------------------------------------------------------
ARCHITECTURE / EXTENSION
--------------------------------------------------------------------------------

  * [v1.1.0 — DONE 2026-05-24] Tool-use as the primary structured-
    output path; multi-provider (Anthropic + Google Gemini).
      Hard cutover. The plain-text branch in ask_llm was deleted; all
      callers go through `llm.ask` with `tools=[SUBMIT_FIX_TOOL]` and
      `tool_choice="submit_fix"`. Anthropic's `strict: true` and
      Gemini's schema-driven function-calling both validate model
      output against the schema server-side, eliminating the Bug 12/
      13 class of validation failures. The architecture reorg split
      `llm/` into errors / response / tools / providers/{anthropic,
      google}, and the ask dispatcher routes on ModelConfig.provider.
      OpenAI integration (originally a separate IMPROVEMENT) is now
      a one-file copy of providers/anthropic.py — drop in
      llm/providers/openai.py, add models to the registry.

  * OpenAI provider — architecture-ready after v1.1.0.
      llm/providers/{anthropic,google}.py demonstrate the shape:
      one `call()` entry point, error mapping, ToolSpec translation,
      Response construction. Adding OpenAI is ~150 LOC of new code
      + tests + an entry in llm/models.py. The dispatcher in
      llm/ask.py picks it up automatically once cfg.provider ==
      "openai" routes there.

  * Multi-file editing.
      Currently `<file_slug>/` directory per file under .refine/.
      Adding multi-file support is additive: refine() takes a list of
      relpaths instead of one, evaluator runs leanforge per file,
      controller tracks per-file state. Spec lists as v2.


--------------------------------------------------------------------------------
OPEN QUESTIONS (decide when next relevant)
--------------------------------------------------------------------------------

  * Sig_changed: soft today (warning + escalate on repeat). Real
    runs should tell us if it over-accepts. If yes, promote to hard.

  * Wall-clock cap.
      We have lean_timeout (per subprocess), cost cap (cumulative $),
      iteration cap (count). No wall-clock cap. A bad LLM response
      could in principle stall the loop for tens of minutes. Add
      --max-wall-seconds?

  * Auto-commit successful runs.
      `git add <file>; git commit -m "refine: fix <file>"` on SUCCESS,
      gated on a flag. Tempting but writes to user's repo state.
      Could surprise; needs a clear opt-in.

  * Output streaming.
      Currently the user waits N seconds in silence between iterations.
      Streaming the LLM's response as it generates would feel more
      responsive. Trade-off: harder to parse JSON incrementally; would
      need a JSON streaming parser.

  * Distinction between "Lean elaboration error" and "edits invalid".
      Both currently end up as failure modes but they're conceptually
      different — one means the LLM proposed bad code, the other means
      the LLM produced bad coordinates. The recent_failures memo should
      probably surface this distinction more loudly.
